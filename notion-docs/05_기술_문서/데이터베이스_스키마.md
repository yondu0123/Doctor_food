## 📋 테이블 상세

### 1. Users (사용자)

사용자 계정 정보를 저장하는 테이블

| 컬럼명 | 타입 | 제약조건 | 설명 |
|--------|------|----------|------|
| `id` | INTEGER | PRIMARY KEY | 사용자 고유 ID |
| `email` | VARCHAR | UNIQUE, NOT NULL | 이메일 (로그인 ID) |
| `username` | VARCHAR | UNIQUE, NOT NULL | 사용자명 |
| `hashed_password` | VARCHAR | NULLABLE | 해시된 비밀번호 (OAuth는 NULL) |
| `full_name` | VARCHAR | | 실명 |
| `phone` | VARCHAR | | 전화번호 |
| `role` | ENUM | NOT NULL | 역할 (admin/caregiver/guardian) |
| `oauth_provider` | VARCHAR(20) | | OAuth 제공자 (email/kakao/naver) |
| `oauth_id` | VARCHAR(100) | | OAuth 사용자 ID |
| `profile_image` | VARCHAR(500) | | 프로필 이미지 URL |
| `is_active` | INTEGER | DEFAULT 1 | 활성화 상태 (0/1) |
| `created_at` | DATETIME | DEFAULT NOW() | 생성일시 |
| `updated_at` | DATETIME | ON UPDATE | 수정일시 |

**인덱스:**
- `idx_users_email` (email)
- `idx_users_username` (username)

**역할 (UserRole Enum):**
- `admin`: 시설 관리자
- `caregiver`: 간병인
- `guardian`: 보호자

---

### 2. Patients (환자)

요양원 입소 환자 정보를 저장하는 테이블

| 컬럼명 | 타입 | 제약조건 | 설명 |
|--------|------|----------|------|
| `id` | INTEGER | PRIMARY KEY | 환자 고유 ID |
| `name` | VARCHAR | NOT NULL | 환자 이름 |
| `birth_date` | DATE | NOT NULL | 생년월일 |
| `gender` | VARCHAR(10) | NOT NULL | 성별 (male/female) |
| `room_number` | VARCHAR(20) | | 병실 번호 |
| `medical_conditions` | TEXT | | 기저질환 (JSON 배열) |
| `dietary_restrictions` | TEXT | | 식이제한 (JSON 배열) |
| `allergies` | TEXT | | 알레르기 (JSON 배열) |
| `guardian_name` | VARCHAR | | 보호자 이름 |
| `guardian_phone` | VARCHAR | | 보호자 연락처 |
| `is_active` | INTEGER | DEFAULT 1 | 활성화 상태 (0/1) |
| `created_at` | DATETIME | DEFAULT NOW() | 등록일시 |
| `updated_at` | DATETIME | ON UPDATE | 수정일시 |

**인덱스:**
- `idx_patients_name` (name)
- `idx_patients_room` (room_number)
- `idx_patients_active` (is_active)

**예시 데이터:**
```json
{
  "medical_conditions": ["당뇨", "고혈압", "치매"],
  "dietary_restrictions": ["저염식", "당뇨식", "연하곤란식"],
  "allergies": ["땅콩", "새우", "복숭아"]
}
```

### 3. Meals (식사 기록)

환자의 식사 사진 및 AI 분석 결과를 저장하는 테이블

| 컬럼명 | 타입 | 제약조건 | 설명 |
|--------|------|----------|------|
| `id` | INTEGER | PRIMARY KEY | 식사 기록 고유 ID |
| `patient_id` | INTEGER | FOREIGN KEY | 환자 ID (→ patients.id) |
| `meal_type` | VARCHAR(20) | NOT NULL | 식사 유형 (breakfast/lunch/dinner/snack) |
| `meal_date` | DATE | NOT NULL | 식사 날짜 |
| `image_url` | VARCHAR(500) | NOT NULL | 식사 사진 경로 |
| `ai_analysis` | TEXT | | AI 분석 결과 (JSON) |
| `notes` | TEXT | | 특이사항 메모 |
| `created_at` | DATETIME | DEFAULT NOW() | 등록일시 |
| `updated_at` | DATETIME | ON UPDATE | 수정일시 |

**인덱스:**
- `idx_meals_patient` (patient_id)
- `idx_meals_date` (meal_date)
- `idx_meals_patient_date` (patient_id, meal_date)

**식사 유형 (MealType):**
- `breakfast`: 아침
- `lunch`: 점심
- `dinner`: 저녁
- `snack`: 간식

**AI 분석 결과 (ai_analysis JSON):**
```json
{
  "foods": ["밥", "된장찌개", "김치", "계란후라이"],
  "calories": 520,
  "protein": 25.5,
  "carbs": 68.0,
  "fat": 12.3,
  "sodium": 890,
  "fiber": 5.2,
  "sugar": 8.5,
  "portion_consumed": "완식",
  "dietary_compliance": true,
  "warnings": [],
  "recommendations": ["나트륨 섭취 주의"]
}
```

### 4. NutritionSummaries (영양소 일일 요약)

환자별 일일 영양소 섭취량을 집계하는 테이블

| 컬럼명 | 타입 | 제약조건 | 설명 |
|--------|------|----------|------|
| `id` | INTEGER | PRIMARY KEY | 요약 고유 ID |
| `patient_id` | INTEGER | FOREIGN KEY | 환자 ID (→ patients.id) |
| `date` | DATE | NOT NULL | 날짜 |
| `total_calories` | FLOAT | DEFAULT 0 | 총 칼로리 (kcal) |
| `total_protein` | FLOAT | DEFAULT 0 | 총 단백질 (g) |
| `total_carbs` | FLOAT | DEFAULT 0 | 총 탄수화물 (g) |
| `total_fat` | FLOAT | DEFAULT 0 | 총 지방 (g) |
| `total_sodium` | FLOAT | DEFAULT 0 | 총 나트륨 (mg) |
| `meal_count` | INTEGER | DEFAULT 0 | 식사 횟수 |
| `created_at` | DATETIME | DEFAULT NOW() | 생성일시 |
| `updated_at` | DATETIME | ON UPDATE | 수정일시 |

**인덱스:**
- `idx_nutrition_patient` (patient_id)
- `idx_nutrition_date` (date)
- `idx_nutrition_patient_date` (patient_id, date) - UNIQUE

**제약조건:**
- UNIQUE (patient_id, date): 환자별 날짜당 하나의 요약만 존재

## 🔗 관계 (Relationships)

### 1. Users → Patients (1:N)
- 한 명의 사용자(간병인)가 여러 환자를 담당할 수 있음
- 현재는 명시적 FK 없음 (향후 추가 예정)

### 2. Patients → Meals (1:N)
- 한 환자가 여러 식사 기록을 가질 수 있음
- `meals.patient_id` → `patients.id`
- ON DELETE CASCADE: 환자 삭제 시 식사 기록도 삭제

### 3. Patients → NutritionSummaries (1:N)
- 한 환자가 여러 날짜의 영양소 요약을 가질 수 있음
- `nutrition_summaries.patient_id` → `patients.id`
- ON DELETE CASCADE: 환자 삭제 시 영양소 요약도 삭제

## 🔍 주요 쿼리 예시

### 1. 환자의 최근 7일 식사 기록 조회
```sql
SELECT * FROM meals
WHERE patient_id = 1
  AND meal_date >= DATE('now', '-7 days')
ORDER BY meal_date DESC, created_at DESC;
```

### 2. 환자의 주간 영양소 통계
```sql
SELECT 
  date,
  total_calories,
  total_protein,
  total_carbs,
  total_fat,
  total_sodium,
  meal_count
FROM nutrition_summaries
WHERE patient_id = 1
  AND date >= DATE('now', '-7 days')
ORDER BY date DESC;
```

### 3. 오늘 식사하지 않은 환자 목록
```sql
SELECT p.* FROM patients p
WHERE p.is_active = 1
  AND NOT EXISTS (
    SELECT 1 FROM meals m
    WHERE m.patient_id = p.id
      AND m.meal_date = DATE('now')
  );
```

### 4. 식이제한 위반 식사 조회
```sql
SELECT * FROM meals
WHERE patient_id = 1
  AND json_extract(ai_analysis, '$.dietary_compliance') = 0
ORDER BY meal_date DESC;
```
